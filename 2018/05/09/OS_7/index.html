死锁

发表于 2018-05-09 | 分类于 操作系统

前言

目标

  1. To develop a description of deadlocks, which prevent sets of concurrent processes from completing their tasks.

  2. To present a number of different methods for preventing or avoiding deadlocks in a computer system.

初步了解

死锁发生必要条件 
1. 互斥

Multual exclusion: only one process at a time can use a resource.

2. 占有且等待（占有某资源的同时在等待别的资源）

Hold and wait: a process holding at least one resource is waiting to acquire additional resources held by other processes.

3. 非抢占（资源只能在进程完成任务后自动释放，而不能被其他进程抢占）

No preemption: a resource can be released only voluntarily by the process holding it, after that process has completed its task.

4. 循环等待（互相等待对方占有的资源，导致成环的现象出现）

ps.必须四个同时满足才会发生死锁

系统模型 System Model

系统描述

  • 资源类型 $R_1, R_2, \cdots, R_m$

    • 内存空间
    • CPU周期
    • 文件
    • I/O 设备

  • 每个资源类型 $R_i$ 有 $W_i$ 个实例

  • 使用资源的正常顺序

    • request 申请（如果申请不被立即允许，则会陷入等待）
    • use 使用
    • release 释放

资源分配图 Resource-Allocation Graph

  1. 定义：顶点 Vertex 和 边 Edge 的有向图

  2. 详细：

    1. 顶点分为两类 进程（圆形） 和 资源（矩形），边也分为两类 申请边 和 分配边，按照使用资源的顺序，申请边会变为分配边
    2. 资源（矩形）中每个圆点代表一个实例

注意：

  1. 环中每一类资源如果只有一个实例时，会发生死锁，如果有多个资源的话，则是有可能发生死锁。
  2. 如果在图中，有进程在使用完资源后可以释放该资源，解开环的话，就不是死锁状态，死锁状态是不能解开的。

死锁处理

分类

  1. Ensure that the system will never enter a deadlock state.

  2. Allow the system to enter a deadlock state and then recover.

  3. Ignore the problem and pretend that deadlocks never occur in the system; used by most operating systems, including UNIX.

死锁预防 Deadlock Prevention

是一组方法，从四个必要条件去解决死锁问题，保证死锁不会发生（第一类处理方法），通过限制申请资源去预防死锁

  1. Mutual Exclusion：可以令资源都为共享去避免，但是通常不可行，因为有些资源只能是非共享的

  2. Hold and Wait：must guarantee that whenever a process requests a resource, it does not hold any other resources

    1. 分类：
      1. 进程在执行前必须申请并获得所有资源（这样的话，它也可以不在申请一个资源的时候还占着别的资源，因为在这个系统是非抢占时，只有当进程运行完时才会释放资源）
      2. 进程在释放完自己的资源后才可以申请资源
    2. 缺点：Low resource utilization and starvation possible
    3. 好处：死锁现象不会发生

  3. No preemption：当一个资源占有资源并在等待别的资源，那么它现在占有的资源都可被抢占，只有当它重新获得原有资源和申请的资源的时候，才可以重新执行

    1. 缺点：不适用于打印机或者磁带等

  4. Circular wait：impose a total ordering of all resource types, and require that each process requests resources in an increasing order of enumeration

死锁避免 Deadlock Avoidance

  1. 定义：根据每个进程可能的申请每种资源类型实例的最大需求的事先信息，构造出一个算法以确保不会进入死锁状态

  2. 资源分配状态 Resource-allocation state 是由可用资源 the numebr of available resources 和已分配资源 the number of allocated resources，及进程最大需求 the maximum demands of the processes 所决定

安全状态

  1. 安全状态算法：When a process requests an available resource, system must decide if immediate allocation leaves the system in a safe state.

  2. 安全状态：A sequence <$P_1,P_2,\cdots,P_n$> that for each $P_i$, the resources that $P_i$ can still request can be satisfied by currently available resources + resources held by all the $P_j$, with j < i. 即 $P_1$ 能申请的资源等于系统现在空余的资源，所以他申请，用完后释放掉，没人可以困住它， $P_2$ 则可以申请的资源等于 $P_1$ 目前占有的资源加上系统空余的资源，那么最不济的情况当 $P_1$ 释放后，也不会被困住

  3. 注意：

    1. If a system is in safe state => no deadlocks
    2. If a system is in unsafe state => possibility of deadlock.
    3. Avoidance => ensure that a system will never enter an
      unsafe state.

  4. 缺点：资源使用率更加低，因为有时候明明有空缺的资源，为了安全都不能被使用

资源分配图算法

  1. 面对于单实例的情况

  2. 需求边 request edge（虚线表示），会转变成分配边

  3. 定义：The request can be granted only if converting the request edge to an assignment edge does not result in the formation of a cycle in the resource allocation graph

  4. 算上需求边，如果没有环的出现，则是安全状态

  5. 需要$n^2$次操作，其中n是系统的进程数量（利用图求环的算法）

银行家算法

  1. 出现原因：资源分配图算法对于多实例情况不适用

  2. 结构：

    1. n是进程的数量，m是资源的种类数量
    2. Available：Vector of length m，Available[i]=k代表$R_i$具有k个可用的实例
    3. Max：n×m 矩阵，Max[i][j] = k代表进程 $P_i$ 需要最多k个$R_j$
    4. Allocation：n×m 矩阵，Allocation[i][j] = k代表 $P_i$ 现在有k个 $R_j$
    5. Need：n×m 矩阵，Need[i][j] = Max[i][j] - Allocation[i][j]

  3. 资源分配算法 Resource-Request Algorithm

    1. 用处：确定该请求是否安全
    2. 定义：Request = request for process，Request[i][j] = k代表$P_i$ wants k instances of resource type $R_j$
    3. 步骤：
      1. If Request[i] $\leq$ Need[i] go to step 2. Otherwise, raise error condition,since process has exceeded its maximum claim.
      2. If Request[i] $\leq$ Available[i], go to step 3. Otherwise Pi must wait, since resources are not available.（没有可用资源）
      3. Pretend to allocate requested resources to Pi by modifying the state as follows:
        1. Available = Available – Request;（满足需求）
        2. Allocation[i]= Allocation[i] + Request[i];（分配资源）
        3. Need[i] = Need[i] – Request[i];（进程剩余所需）
        4. If safe => the resources are allocated to $P_i$.
        5. If unsafe => $P_i$ must wait, and the old resource-allocation state is restored

  4. 安全性算法 Safety Algorithm

    1. 用处：确定计算机系统是否处于安全状态
    2. 定义：找到一个顺序，最后可以让所有的进程都运行结束，释放完全部资源（这个顺序中，分配和释放资源的过程要满足资源分配算法）

死锁检测 Deadlock Detection

  1. 原因：如果不采用死锁预防和死锁避免算法，可能会出现死锁，这个时候系统应该提供：
    1. 检测系统是否发生死锁现象的算法
    2. 从死锁状态中恢复算法

单实例情况

  1. 等待图 wait-for graph

  2. 利用资源分配图删除资源点后可以得到等待图

  3. 当存在环的时候，系统存在死锁，检查环的算法需要$n^2$的时间复杂度

多实例情况

  1. 检测方法：使用银行家算法后，要是不能清空，那么剩下的进程都在死锁状态,需要$O(m \times n^2)$的时间复杂度

调用

  1. 决定因素：
    1. 死锁发生的频率：如果经常发生死锁，则需要经常调用死锁检测
    2. 死锁发生受影响进程的数量

死锁恢复 Recovery from Deadlock

进程终止

  1. 分类：
    1. 终止所有死锁进程
    2. 一次只终止一个进程直到取消死循环为止：需要考虑以什么的顺序去取消，比如进程的优先级、进程已经运行的时间等

资源抢占

  1. 定义：从进程中抢占资源给其他进程使用

  2. 需要考虑的问题：

    1. 选择一个牺牲品 Selecting a victim：minimize cost.
    2. 回滚 Rollback：return to some safe state, restart process for that state.
    3. 饥饿 Starvation：same process may always be picked as victim, include number of rollback in cost factor.
