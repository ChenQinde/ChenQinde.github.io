进程同步

发表于 2018-05-09 | 分类于 操作系统

背景

使用 count 去解决生产者消费者问题时，如果生产者进程和消费者进程并发执行，则有可能无法保持数据一致性。因为两个程序都有自己的局部寄存器，当要改变 count 时，他们会取出 count 到各自的寄存器中，然后修改，再存进内存中，这样会使数据和与其不同。

定义：以上现象定义为竞争条件 race condition，即多个进程并发访问和操作同一数据且执行结果与访问发生的特定顺序有关。

临界区问题

  1. 定义：设计一个以便进程协作的协议

    • 进入区 entry section：请求允许进入其临界区
    • 临界区 critical section：可能改变共同变量、更新一个表、写一个文件
    • 退出区 exit section：退出临界区要执行的代码
    • 剩余区 remainder section

  2. 要求：

    • 互斥 Mutual Exclusion：只有一个进程可以在临界区执行
    • 前进 Progress：没有进程在临界区时，应允许一个需要进入临界区的进程进入
    • 有限等待 Bounded Waiting：不允许需要进入临界区的进程被无限延迟

  3. 处理操作系统中的临界区问题：

    • 抢占内核：允许处于内核模式的进程被抢占
    • 非抢占内核：处于内核模式的进程会一直运行，直到退出内核模式、阻塞才会轮到别的进程。

硬件同步

  1. 对于单处理器环境，只要在修改共享变量时禁止中断出现便可，但对于多处理器环境，禁止中断会很耗时，降低系统效率，所以许多现代计算机系统提供了特殊硬件指令以允许原子性检查不可中断指令的进行。

  2. 抽象出来的代码：

    • TestAndSet：设置一个全局变量 lock ，当一个进程进入 critical section 时，将 lock 锁住，当进入 exit section 时，将 lock 解开
    • Swap：设置一个全局变量 bolt ，当一个进程进入 critical section 时，将 bolt 中的 0 和 进程的 key = 1 交换，因为全局只有一个 0，所以可以保护共享内存区

  3. 局限：以上算法并没有解决有限等待的问题

  4. 改进：

  1. 忙等待 \ 自旋等待现象 busy waiting

  2. 优点：不用进行上下文等待

  3. 缺点：浪费CPU时间

信号量 Semaphore

  1. 好处：让等待的进程不会处于忙等待，故可以不占用CPU时间

  2. 缺点：需要进行上下文切换

  3. 操作：

    • semaphore：一个进程在某一特殊点上被迫停止执行直到接收到一个对应的特殊资源值，这个资源值称为信号量
    • wait() or acquire()：当进程需要使用资源时，信号量减少（P操作）

    • signal() or release()：当进程释放资源时，信号量增加（V操作）

      ps. 两个操作都是原子操作

用法

  1. 结构：

    • waiting queue：每个 entry 中有两个 item，一个是 value(of type integer)，另一个是指向下一个 entry 的指针
    • block操作：将进程加入到 waiting queue
    • wakeup操作：将进程从移除 waiting queue，并放入ready queue

  2. 分类：

    1. counting semaphore：值域无限
    2. binary semaphore：0 或者 1

  3. 注意：wait() 和 signal() 未必要一起出现，比如现在有两个进程，一个需要先执行，一个需要后执行，可以在前者中放入 signal() ，在后者中放入 wait()

实现

  1. 注意：wait() 和 signal() 需要是原子性操作，所以我们仍需要忙等待，不过忙等待限制在 wait() 和 signal() 的临界区中，这些区比较短，而如果令忙等待像以前那样，存在于应用程序中，临界区可能被占用很长时间，这样极为低效。
