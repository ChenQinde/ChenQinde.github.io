An example is the root-to-leaf path 1->2->3 which represents the number 123.

Find the total sum of all root-to-leaf numbers.

Note: A leaf is a node with no children.

题目理解

分别将从根到每条叶节点的路径上所经过的数字组合在一起（处在树中越高层的在数字中的越高位），最后将这些数字加起来，输出结果。

解析

使用DFS，每次搜索下一个点时都会存储一个临时值（该点的父节点到根所组成的数字），接着当遇到了叶节点（没有任何子节点的点）时，将临时值乘以10加上叶节点的值，最后返回回去，将所有DFS得到的结果加起来便是答案。

有关比特币你可能不知道的地方

前言

了解比特币也有一段时间了(其实就两个星期吧)，打算开始写一系列有关比特币的博客，这一系列的会包含许多比特币中的难点，也会分析比特币Go版的源码，可能会有写错或者理解错的地方，很希望有人会看我写的博客并提出来这些错误。

引出问题

由于比特币的历史账本十分多，占用内存也很多，所以对于普通用户而言，为了验证自己的交易，需要下载整条链到本地显然不太现实，故在BIP:37引入一种技术称为 simplified payment verification(SPV)。

SPV简单来说就是只存储区块头，当需要某个交易的信息时，再问附近的重节点（存储有整条链的用户）拿包含该交易的块，这样既能拿到所需信息，又能防止重节点有欺诈行为（因为有区块头可以验证，所以无法伪造）。

但如果经常直接向重节点要区块，会存在隐私泄露的问题，因为重节点可以通过多个区块中交易的交集来推测出你的地址。

Bloom Filter

定义：Bloom Filter 是一种基于概率的数据结构，用来判别一个元素是否在一个集合中，但是会有一定的误判率，因为它只能够告诉你这个元素究竟是不在集合内还是可能在集合内。(就是因为它无法说明某个元素是否一定在集合内，所以会出现误判率)

解释：下图有一个大小为18的数组(初始化每位为0)，h1、h2、h3是三个不同的哈希函数(输出为0到17)，我们有一个集合 {x, y, z}，将x值分别输入h1、h2、h3，得到三个数字，将数组中对应的位置变为1，接着以相同方式，将y、z值都也输入h1、h2、h3，并将对应位置变为1。将数组和h1、h2、h3传给别人，别人就可以判断某个元素ω是否在集合中(图中的ω就不属于这个集合，因为它有个哈希值是1与数组中的不匹配)

缺点：

  1. 误报：因为该filter无法说明某个元素是否一定在集合内
  2. 禁止删除：如果允许删除某个元素，会影响整个filter，因为这个被删除的元素有可能与别的元素有重叠的部分(可通过将数组中的每个元素变为计数器，但这样会浪费大量空间)

背景

  1. 虚拟内存定义：在具有层次结构存储器的计算机系统中，采用自动实现部分装入和部分对换功能，为用户提供一个比物理主存容量大得多的，可寻址的一种“主存储器”。

  2. 好处：

    1. 在主存中保留多个进程：由于对任何特定的进程都仅仅装入它的某些块，因此就有足够的空间放置更多进程。
    2. 进程可以比主存的全部空间还大：通过分段或分页的虚拟内存，程序员可以获得巨大的主存，大小与磁盘储存器相关。
